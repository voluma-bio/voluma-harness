# Research: Future Directions, Monitor Gaps, and the Security Frontier

**Scope.** This is an adversarial research memo, not a replacement design. It reads the proposed [monitor framework](../../codebase-audit-rewrite/design/monitor-framework.md), [capability caps](../../codebase-audit-rewrite/design/capability-caps.md), [surface notes](../surface-architecture-notes.md), and v1 §§7, 9, 10 against the source studies in [`research/`](../research/). “Better” below means a cited system has an explicit control at the named boundary—not that it solves agent safety generally.

## Monitor and security gaps

### 1. “Irreversible actions have sync hooks” is an assumption, not an enforceable coverage contract

**Failure / attack.** A drifting worker uses a permitted shell path to create a remote side effect (for example, starts an external deployment, writes a persistent CI trigger, or sends a credential through a network command) that is not in the hook registry. The parallel reviewer detects the trajectory after the action; broker gating only prevents its *next* model call. This is not hypothetical timing pedantry: OpenAI Agents SDK documents that a parallel input guardrail may lose the race to tool execution, while a blocking guardrail and a tool-input guardrail run before the covered function call. It also explicitly limits normal tool-guardrail coverage—hosted tools, handoffs, and agents-as-tools are outside that pipeline. [OpenAI Agents SDK guardrails](https://openai.github.io/openai-agents-python/guardrails/)

**Where the proposal is weaker.** The framework correctly places single irreversible actions in hooks, but neither the monitor binding nor the capability contract says which *granted* operations must have a synchronous interceptor, whether every harness binding provides one, or what happens when it cannot. That permits a future native-harness/tool addition to silently bypass the premise.

**Position now.** Add a versioned **operation-risk manifest** to the launch bundle: every `ToolGrant` / command / network egress class declares `observe-only`, `preflight-required`, or `not-permitted`. Launch must fail closed when a `preflight-required` operation has no deterministic, binding-specific interceptor. Keep LLM review out of this path; use a schema/policy preflight that can deny before dispatch. The manifest makes the one-turn-lag guarantee auditable rather than aspirational.

### 2. Reviewer input is an untrusted, high-authority control channel

**Failure / attack.** The monitored journal includes tool output, repository text, web content, and model text. A malicious issue, file, or tool response can instruct the reviewer to “approve the deployment,” “kill the worker,” or demand escalation. Because a reviewer receives rich trajectory context and can emit `pause`, `kill`, or `escalate`, it becomes a prompt-injection target with control-plane authority. A worker can also deliberately shape its transcript to cause a weak reviewer to pause a competing worker (availability attack) or to normalize an unsafe action (collusive/reward-hacking failure).

**What prior art establishes.** OpenAI separates deterministic tool-input checks from agent-level input/output checks and exposes an explicit tripwire boundary; a tool guardrail can reject or replace a covered tool input before execution. It does **not** claim that an LLM guardrail is safe merely because it saw untrusted text. [Guardrail workflow boundaries and tripwires](https://openai.github.io/openai-agents-python/guardrails/). LangGraph similarly treats an interrupt as a runtime control-flow pause that must be explicitly resumed from a persisted checkpoint; it is not a separate monitor security domain. [LangGraph interrupts](https://docs.langchain.com/oss/python/langgraph/interrupts)

**Position now.** Split monitors by trust class in the binding policy:

- **Deterministic monitors** may emit `block`, `pause`, or `kill` for narrowly typed evidence (policy match, spend ledger, signed provenance, path rule).
- **Model-judgment monitors** may only emit an evidence-bearing `recommend` / `warn`; a separate deterministic admission policy translates that recommendation into a high-impact verdict.
- Store the reviewed event IDs, policy/model/prompt version, structured evidence, and verdict signature. Treat journal payload as data—not executable monitor instructions—and never let monitor text directly populate an escalation or tool argument.

This does not make LLM judgment trustworthy; it prevents it from being the sole authorization mechanism.

### 3. The broker gate has no durability or availability semantics

**Failure / attack.** A reviewer posts `pause`, then the broker/daemon restarts before the next issuance. If the pause flag, reviewer cursor, and issuance decision are not committed atomically, the worker can resume and obtain a token (fail-open). Conversely, a broker outage or a stuck reviewer can globally freeze work (fail-closed outage). A compromised broker is also a single point that can ignore every monitor and budget decision.

**What prior art establishes.** LangGraph persists the interrupt/checkpoint state and requires an explicit `Command` to resume, including handling outstanding interrupt IDs under a lock; this is stronger recovery semantics than an in-memory “flagged” state. [LangGraph interrupt persistence](https://docs.langchain.com/oss/python/langgraph/interrupts). OpenAI’s SDK makes the less-safe parallel race explicit rather than treating cancellation as retroactive. [Agents SDK execution modes](https://openai.github.io/openai-agents-python/guardrails/#execution-modes)

**Position now.** Make broker admission a durable state machine, not a boolean: `(thread_id, admission_epoch, state, reason_event_id, policy_version, expires_at)`. Commit the verdict and incremented epoch before responding; require a short-lived, audience-bound, one-use model-call lease carrying that epoch. On recovery, a non-terminal `paused` admission remains paused until an authorized clear event is durably written. Define a per-policy availability mode (`fail-closed`, `warn-only`, or `human-resume`) and show it in the UI. This reserves an HA/recovery seam without committing v1 to a distributed broker.

### 4. Monitor work can consume or deny the very budget it is meant to protect

**Failure / attack.** An attacker causes high-entropy or long tool output every turn, inducing expensive deep reviews. Or a misconfigured profile attaches several reviewers that all lag, repeatedly deep-review the same backlog, and hold a worker at a token boundary. The proposed tiering is sensible, but it has no monitor-specific deadline, byte/window limit, reservation, or defined response when the monitor cannot keep up.

**What prior art establishes.** Inspect explicitly bounds evaluation work with task/evaluator limits (including time, samples/epochs, and configured model/token limits), while keeping the scorer separate from the evaluated sample’s execution. [Inspect AI limits](https://inspect.aisi.org.uk/limits.html) and [eval runners](https://inspect.aisi.org.uk/eval-runners.html). AutoGen exposes composable termination conditions such as message, time, and token-use limits at the team runtime. [AutoGen termination](https://microsoft.github.io/autogen/stable/user-guide/agentchat-user-guide/termination.html)

**Position now.** Extend the cap/policy model with a **monitor budget envelope**: maximum concurrent monitors, review window bytes/turns, scorer/deep-review model allowlists, per-monitor and aggregate subtree spend, deadline, and backlog rule. A binding must declare whether overload degrades to `annotate`, `warn`, or `pause`; it must not silently choose. Attribute monitor spend separately *and* include it in the parent work-item reservation, so “safe” execution cannot evade the user’s total ceiling.

### 5. Freezing an agent does not revoke its already-running or persistent effects

**Failure / attack.** Before the next model call is withheld, a tool starts a background process, opens a long-lived remote connection, or writes a malicious project hook/configuration. Pausing a sandbox can preserve that process and its in-memory credentials. A later session, a sibling through a shared volume, or a human/CI process executes the planted persistence without holding the original capability.

**What prior art establishes.** E2B documents that pause preserves filesystem, memory, running processes, and data; timeout can therefore preserve rather than destroy execution state. [E2B persistence](https://e2b.dev/docs/sandbox/persistence). Daytona distinguishes stop (filesystem retained), archive, delete, and ephemeral deletion; its volumes persist independently and can be shared, while its ephemeral sandbox story includes session-token revocation. [Daytona lifecycle](https://www.daytona.io/docs/en/sandboxes/), [volumes](https://www.daytona.io/docs/volumes/), and [security exhibit](https://www.daytona.io/docs/ja/security-exhibit/)

**Position now.** Bind every brokered secret, network connection, background process, sandbox, snapshot, and volume mount to a **capability lease** with expiry and `revocationEpoch`. A `pause` stops new model calls; `kill` revokes leases and terminates/isolates tracked processes; only a completed destruction transition may claim cleanup. Mount/restore/fork must require a new capability decision and preserve provenance. Classify execution-triggering workspace paths (hooks, CI, package scripts, harness/config roots) as a separate write grant, not ordinary source-file write access.

### 6. Capability caps still leave two confused-deputy paths under-weighted

**Failure / attack A — ambient authority.** `workspace-write` (or `danger-full-access`) lets a worker use any unmediated binary, inherited local service, or raw provider credential. The cap then governs only Meridian APIs while the worker acts as the operator elsewhere. The cap document acknowledges this for Tier 1, but the product surface needs to communicate the assurance difference rather than display all runs as equivalently “capped.”

**Failure / attack B — persistence deputy.** A low-privilege worker writes a permitted-but-executable project artifact—`package.json`, a Git hook, a workspace task, an agent/harness configuration, or a shared volume record. A later privileged human, CI job, or sandbox executes it. The original worker has converted scoped write authority into a future principal’s authority.

**What prior art establishes.** Claude Code’s hooks are explicit executable callbacks around tool lifecycle, whereas its sandboxing guidance separately describes filesystem/network control for commands and their subprocesses; a hook decision is not a kernel boundary. [Claude Code hooks](https://code.claude.com/docs/en/hooks-guide) and [sandboxing](https://www.anthropic.com/engineering/claude-code-sandboxing). Codex likewise distinguishes approval workflow from a technical command sandbox with network and path restrictions. [Codex sandboxing architecture](https://developers.openai.com/codex/cloud/environments/) (and its [CLI sandbox command](https://developers.openai.com/codex/cli/reference/#sandbox)).

**Position now.** Add an explicit `enforcement_assurance: cooperative | sandboxed | remote-authoritative` to every spawn and audit record; do not call `cooperative` a security boundary. Make mediated broker authentication the default for managed harnesses (raw provider keys never materialize in worker environments), bind a bearer credential to spawn, audience, expiry, and revocation epoch, and require a separate `policy-write`/`persistent-mount-write` grant for executable configuration and durable mounts. The current surface decision to keep config writes off worker MCP is necessary but insufficient: workspace files and volumes need the same classification.

## Future directions, ranked for Meridian

Ranking weighs product leverage, evidence of real adoption, and how cheaply the current contracts can preserve the option. It intentionally does **not** recommend building every item now.

| Rank | Direction | Evidence the direction is real | Why it fits Meridian | Cheap contract move now |
|---|---|---|---|---|
| **1** | **Journal-backed evaluation and replay** | LangGraph checkpoints state per super-step and exposes history/time travel; LangSmith evaluates traces both offline and online, including human, code, and LLM judges; OpenAI Evals uses versioned eval definitions and sample data. [LangGraph persistence](https://docs.langchain.com/oss/python/langgraph/persistence), [LangSmith evaluation](https://docs.langchain.com/langsmith/evaluation-concepts), [OpenAI Evals](https://github.com/openai/evals) | Meridian is already paying for an event journal, prompt snapshots, launch bundles, CRDT writes, and spans. Without a replay/eval contract those are only forensic logs; with one they become regression cases for harness changes, monitor quality, prompt packages, and write-policy changes. | Reserve immutable `run_manifest` and `evaluation_result` records: normalized event order/IDs and causal parents; agent/bundle/skill/prompt hashes; harness/model/provider parameters; cap/policy and sandbox-image versions; input workspace/artifact hashes; tool request/result recordings; usage/cost; and evaluator/dataset version. Define three honest modes now: **trace replay** (render recorded facts), **simulation replay** (recorded tool results), and **rerun** (new nondeterministic model execution). Do not promise bit-for-bit LLM determinism. |
| **2** | **Budget governance across the agent tree, monitors, and tools** | LangSmith has a concrete per-run/span usage and `total_cost` data contract; its online evaluation also treats whole threads as the quality unit. It does not itself make spend authorization hard-stop, which is precisely the missing control. [LangSmith run data](https://docs.langchain.com/langsmith/run-data-format), [cost tracking](https://docs.langchain.com/langsmith/cost-tracking), [thread-level evaluation](https://docs.langchain.com/langsmith/evaluation-concepts) | The design names static spend ceilings and monitor cost attribution, but the shown `PermissionSet` has no budget grant and the broker is the natural per-model-call admission point. Parallel agents plus parallel reviewers make a per-thread display total inadequate: reservation must be tree-aware. | Add a monotonic `BudgetGrant` alongside `SpawnCapability`: currency/units, parent reservation, child allocatable ceiling, monitor reserve, tool/network reserve, and policy on unknown price/retry. Require the broker to reserve before issuance and settle recorded actual usage afterwards. Keep provider-pricing adapters behind the broker; the durable contract is the reservation/settlement ledger, not a price table. |
| **3** | **Interoperable remote-agent and client sessions** | A2A has a normative specification and protobuf schema for `AgentCard`, tasks, messages, artifacts, lifecycle states, SSE/webhook updates, capabilities, and extensions. ACP v1 separately standardizes interactive editor-to-coding-agent communication with version/capability negotiation and published schemas/SDKs. [A2A specification](https://github.com/a2aproject/A2A/blob/main/docs/specification.md), [A2A protobuf](https://github.com/a2aproject/A2A/blob/main/specification/a2a.proto), [ACP repository](https://github.com/agentclientprotocol/agent-client-protocol) | The proposed query/spawn MCP split already gives Meridian an internal control plane. A2A would let a remote/subcontracted agent participate without pretending it is a local child process; ACP offers a route for IDE clients. Neither replaces CRDT document collaboration or cap enforcement. | Separate **agent identity/descriptor** from harness launch: stable agent and task/context IDs, ordered update IDs, typed artifacts, terminal reasons, negotiated protocol/capability versions, and explicit auth/provenance fields. Make remote tasks a distinct trust tier from local child spawns. Map outward later; do not bake A2A/ACP payloads into core tables or assume either standard carries Meridian’s delegation semantics. |
| **4** | **Durable knowledge with scope, provenance, and retrieval audit** | LangGraph separates thread checkpoints from a cross-thread Store whose records have namespace, key/value, and optional index fields; its docs explicitly flag checkpoint-storage growth and support distinct backend persistence. [LangGraph memory](https://docs.langchain.com/oss/python/langgraph/add-memory), [persistence](https://docs.langchain.com/oss/python/langgraph/persistence) | CRDT documents, journals, and mars catalogs retain artifacts, but the design has no contract for reusable task knowledge, its authority, expiry, or why it was injected. Multi-agent review will otherwise create hidden prompt-memory behavior and stale instructions. | Reserve a `KnowledgeRecord`/retrieval event shape: namespace and scope (user/project/work/thread), origin event/artifact, author and cap provenance, content/version/hash, created/expiry/review status, ACL, index/embedding version, and retrieval citation. Keep memory read-only to agents by default and require explicit retrieval events; never auto-inject mutable memory into the frozen system prompt. |

### Why these four, not “multiplayer sessions” first

Multiplayer editing is already central to the CRDT/local-first trajectory, and A2A’s `contextId` is only task grouping—not conflict resolution or shared-memory collaboration. The more urgent missing leverage is turning Meridian’s existing durable journal into a controlled learning loop, then ensuring the associated monitor and delegation cost is governable. Agent interoperability should reserve IDs, lifecycle, artifacts, and negotiation now because those fields are cheap; it should not force a premature protocol adoption. [A2A task/context model](https://github.com/a2aproject/A2A/blob/main/docs/specification.md)

## Recommendation

**Under-invested direction: journal-backed evaluation and replay.** The design is unusually well positioned: it already requires persist-before-observe events, prompt/context snapshots, versioned mars launch bundles, capability records, document journals, and OTel-style spans. Those are exactly the inputs trace/eval systems preserve to compare an execution against a dataset and evaluator; LangGraph’s checkpoint/time-travel model and LangSmith’s offline/online trace evaluation show the pattern is established, while OpenAI Evals demonstrates versioned evaluator/data artifacts. [LangGraph persistence](https://docs.langchain.com/oss/python/langgraph/persistence), [LangSmith evaluation](https://docs.langchain.com/langsmith/evaluation-concepts), [OpenAI Evals](https://github.com/openai/evals) The cheap move is schema provenance and explicit replay classes now—not an eval product. It makes later monitor calibration, regression detection, post-incident investigation, and safe harness migration possible; without it, the monitor framework will accumulate expensive judgments that cannot be reproduced, measured, or improved.
