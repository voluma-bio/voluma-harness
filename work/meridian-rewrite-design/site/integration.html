Integration & roadmap

v1 keeps Mars as the prompt-package authority, treats cloud sync as an optional Yjs peer, and preserves the agent-facing contracts that make the rewrite safe to adopt.

Boundary decision: the daemon owns local runtime state and collaboration; Mars owns prompt-package resolution and launch policy. Cloud replication never becomes a control plane.

Mars integration

mars-agents stays Rust and stays the prompt package manager. v1 does not reimplement package resolution, lock ownership, target lowering, model aliases, or launch-bundle construction.

Contract

  • voluma mars <args> passthrough with --root injection and VOLUMA_MANAGED=1
  • Bundled or sibling mars binary resolution before PATH
  • mars build launch-bundle --json schema v3 as launch policy authority
  • Agent catalog from .mars/agents/, not harness-native target directories
  • Mars exit codes 0–3 propagated unchanged

What simplifies: v1 only needs Claude Code native target materialization plus canonical .mars/. The five harness targets remain until Mars changes.

Cloud sync (optional)

Local SQLite and Yjs remain authoritative when offline. Cloud sync is a peer, not a controller.

Sync scope

Syncable Local-only
Yjs document updates and checkpoints, document metadata, thread and turn events, optional event summaries Secrets, raw environment snapshots, absolute paths, sandbox and process details, private prompt paths

Protocol: meridian-flow’s Yjs WebSocket at the /ws/yjs prefix. Room names map to document IDs. The local daemon acts as a Yjs peer, not a remote database client.

Prior art synthesis

The studied systems agree on authority boundaries, structured streams, and durable local state. v1 adopts those patterns without surrendering local-first collaboration.

Pi

Two-level provider seam, wire protocol versus vendor. Pure event-driven agent loop. I/O mode adapters over a shared session.

OpenCode

Per-request instance routing via header or query. SSE with register-before-stream. Effect-TS plus Drizzle on SQLite WAL.

Claude Agent SDK

NDJSON stream protocol. Permission interception hooks (canUseTool). Session store dual-write. Agent as declarative data.

CopilotKit

Runtime bridge between web UI and agent. Streaming via server-sent events. Frontend-registered actions as tools.

LangGraph

Graph-based orchestration with checkpoints. State machine for control flow. Persistence adapters for multiple backends.

Omnigent

Meta-harness via executor.harness. Runner and server split. Stateful policy, including cost caps and access controls. Child sessions are first-class objects.

Common patterns adopted

  • Authority daemon + thin clients (Pi, OpenCode, Omnigent) → daemon + CLI/MCP/web model
  • Harness abstraction layer (Pi, Omnigent, AI SDK) → two-class harness
  • SQLite WAL as local state (OpenCode) → data model
  • Structured event stream (OpenCode SSE, Claude SDK NDJSON) → event sink
  • Agent as declarative data (Claude SDK, Pi) → Mars agent catalog
  • Capability/policy enforcement at authority (Omnigent) → capability model
Unique to v1: none of the studied systems have CRDT collaborative editing between human and agent.

Invariant registry

These distinctions make the rewrite explicit: what agents can rely on, what changes behind the boundary, and what is intentionally removed.

Carry agent-facing contracts
  • Spawn statuses: queued, running, finalizing, succeeded, failed, cancelled, timed_out
  • Spawn transition graph and terminal monotonicity
  • Completion report evidence wins over late cancel
  • p<N> spawn IDs and c<N> chat IDs, monotonic
  • Parent/depth: parent_id, MERIDIAN_PARENT_SPAWN_ID, zero-based depth, fail-closed
  • MERIDIAN_CHAT_ID inherited across spawn tree
  • Work attachment precedence: explicit > ambient session > --from inheritance
  • Config merge: CLI flags > environment > YAML/profile > config defaults
  • CLI command paths: spawn, session, work, context, mars, doctor
  • Spawn refs: @latest, @last-failed, @last-completed, p123
  • Fork policy: --fork identity-locked; --fork-fresh allows changes
  • Mars passthrough, launch-bundle v3, filesystem contract
Redesign internals changing
  • Runtime persistence: JSON files → SQLite tables
  • Harness layer: five adapters → Claude Code native + unified API
  • Terminal event extraction: explicit API events replace per-harness classifiers
  • Session transcripts: structured events and turns replace raw history.jsonl
  • Control socket: per-spawn sockets → daemon RPC
  • Observability: structlog → event sink + OpenTelemetry
  • File authority: SQLite/Yjs/daemon replace files-as-authority
  • Agent-edit adapters: Postgres/Hocuspocus → local SQLite/in-process
  • Capability model: D8 audit → enforced runtime
Break intentionally dropped
  • Files-as-runtime-state (spawns/<id>/state.json)
  • history.jsonl as transcript authority
  • Native Cursor, Pi, OpenCode, and Codex adapters
  • Global spawns.jsonl and legacy telemetry JSONL
  • Prompt body sidecar files → system prompt snapshots in SQLite
  • Per-harness target sprawl in default templates

Build order

Start at the authority seam, then add the durable state that allows every later surface to share it.

  1. Prove the write seam — write CLI/MCP → daemon → AgentEditCore → stdout
  2. SQLite journal + Y.Doc coordinator — UpdateJournal/ReversalStore on SQLite
  3. File tracking + source codec — line-oriented with exact round-trip
  4. Core spawn/session/thread/work schema — SQLite tables, CLI daemon client
  5. Claude Code native launch — deny rules, MCP write tool, stream parsing
  6. Webapp — live editor + dashboard over the same daemon
  7. Unified API harness — in-process tools, embeddable agents
  8. Cloud Yjs peer sync — optional, connect as a Yjs peer to flow cloud
