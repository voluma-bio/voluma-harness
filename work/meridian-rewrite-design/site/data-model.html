Voluma v1 data model

Everything is a thread. Each thread is an independent execution context with its own agent configuration. ~36 tables across 7 domains.

Synthesized from 5 systems: Voluma, Meridian-Flow, meridian-cli, OpenCode, Pi. Work context and knowledge base are the same model — a work item is a scoped knowledge container.

Thread & execution

7 tables

Threads own conversation, runtime state, lineage, and their append-only journal. A spawn is a subagent thread — not a separate persistence model. Durable prompt admission (from OpenCode) separates "user submitted" from "prompt was executed." Config lives as columns on the thread row (config_hash, config_snapshot, config_updated_at) — divergence detection, not audit history. No separate epochs table.

Knowledge hierarchy

The work IS the context. When a thread operates within a work: it reads project-level context (architecture, conventions) + work-level context (decisions, handoffs from other threads). It writes new knowledge back to the work's context scope. "Creating a work item" = "creating a new knowledge scope."

Agent & configuration

4 tables

Agents are reusable templates with skills. Skills are typed behavior units (principle, guardrail, reference). Agents can delegate to subagents. Config is snapshotted onto the thread at creation (config_hash, config_snapshot); /reload updates the snapshot in place.

Billing

4 tables

Hybrid model: Stripe subscriptions provision credit lots each period. Users can also purchase one-off lots. Lots are consumed FIFO. Millicredits (1 credit = 1000) give fine-grained metering without floating point. A "debt" lot handles overage without hard-stopping mid-turn.

Identity & workspace

2 tables

Users own workbenches. A workbench is a user's workspace within a project — UI state, billing context, and access-scoped resources. Threads do not belong to workbenches (they belong to projects, shared across collaborators); workbench_id remains only where per-user or per-billing scoping matters: results, credit usage, credit transactions, protected artifacts.

Thread scratchspace via primary work

Each thread's primary work (enforced by thread_works.is_primary unique constraint) provides its folder. The slug creates a per-thread namespace within the work directory:

<work-root>/<work-slug>/
  __status.json           # work metadata
  prompts/                # shared work prompts
  handoffs/               # shared handoffs
  threads/
    <thread-slug>/        # per-thread scratchspace
      artifacts/
      scratch/
      report.md

Threads share work items — multiple threads read and write to the same work's knowledge scope. The thread-level namespace prevents artifact collision without fragmenting knowledge. Threads without a work item (casual chats) get no folder.

Resolved design decisions

Hybrid
Billing model?
Subscriptions provision credit lots each period. Users can also buy one-off lots. Lots consumed FIFO. Millicredits avoid floating point. "Debt" lot handles overage without hard-stopping mid-turn.
Document-backed
Results / artifacts?
Results ARE documents with promotion provenance. Links to Yjs restore point for exact state at capture time. Full thread → turn → tool_call provenance chain. Replaces meridian-cli's simpler per-spawn artifacts.
API client
CLI ↔ DB bridge?
CLI becomes a Voluma API client. Postgres is authoritative. Local write-ahead buffer for offline/crash recovery replays on reconnect. File-based state goes away for the core path.
Sequence
Multi-model responses?
model_responses.sequence per turn. Retries, tool continuations, and fallback chains all produce multiple responses. UI shows final state; debug view shows all responses with latency and cost.
Auto-merge
Document conflicts?
Yjs auto-merge is the default. thread_documents tracks which threads edit each document. Concurrent edits trigger a system-reminder notification. Escalate to user only if validation detects semantic breakage.
Unified
Work = knowledge base?
KB/context and work management are the same model. A work item is a scoped knowledge container. Context sources are scoped to project OR work (exactly one). Validated by Vercel v0's workspace-first architecture — no separate vector DB needed.

Reload flow

Two mechanisms. Notification is cheap and cache-preserving — a system-reminder injected on hash divergence. /reload is explicit, rebuilds the system prompt, and intentionally breaks the cache. Forced reinitialization was vetoed — it destroys prompt cache and makes users pay more.

Prior art: ecosystem patterns

Research across 8 systems reveals a consistent decomposition. "Everything is a thread" is directionally correct — but the thread splits into three primitives that our model maps cleanly.

The three primitives

Every system splits the problem the same way, even if they name it differently:

PrimitiveWhat it holdsExamplesOur v1 equivalent
Message containerOrdered transcript, append-onlySlack thread, OpenAI thread, Anthropic client historythreads → turns → turn_blocks
Execution bindingRuntime state, lifecycle, coordinationOpenAI run, CrewAI kickoff, AutoGen team cyclethread.status + spawn_status + event_journal
Configuration identityReusable agent/behavior templateOpenAI assistant, CrewAI agent, LangGraph context_schemaagent_definitions + skills + agent_skills

The thread is the JOIN point — it references config (agent_slug, config_hash), contains state (turns, active_leaf_turn_id), and tracks execution (status, spawn_status). Three concerns on one entity as columns/relations, not one blob.

Why we don't need a separate Run table

OpenAI separates Thread from Run because one thread has multiple runs — each user message triggers a new run against the same conversation. Our model differs: a thread IS a run. It executes continuously until completion or cancellation. Multiple turns within a thread aren't separate runs — they're the thread continuing. Re-execution means a new thread (fork), not a new run on the same container.

This works because our threads are agent-scoped (one thread, one agent, one execution context). OpenAI's threads are user-scoped (the user's conversation across different assistants).

Thread taxonomy across systems

Slack
Flat transcript with root marker. thread_ts identifies root, replies array lists participants. NOT recursive — flat log with thread markers. Closest to our turns within a thread.
Discord
Child channel with parent pointer. Threads are separate channel objects with parent_id, metadata, membership. Stronger hierarchy than Slack. Thread membership modeled separately.
IMAP RFC 5256
Message graph from headers, reconstructed into tree. References chain + In-Reply-To. THREAD REFERENCES algorithm builds parent/child tree. Cleanest recursive model in the ecosystem.
OpenAI Assistants
Thread + Assistant + Run triple. Thread = persistent conversation container. Assistant = reusable config. Run = execution binding. No parent/child threads — flat transcript container.
Anthropic Messages
Explicitly stateless. No server-side thread — "thread" is client-managed history sent each request. Mid-conversation system messages with equal authority. Validates our thread-level config snapshot approach.
LangGraph
Typed state bag + per-key reducers + immutable context. add_messages merges by ID. context_schema for read-only ambient data, state_schema for mutable shared state. Maps to our working_state + context_sources.
CrewAI
Declarative crew/task graph + typed workflow state. Crew-level defaults override agent-level. Tasks carry context dependencies. Flows use Pydantic state models mutated in handlers.
AutoGen
Nested session tree of agent states. Team state contains per-agent states + manager state. HandoffMessage for delegation. Closest to a persisted multi-agent thread tree in the ecosystem.
Open question: structured handoff targets

AutoGen's Swarm pattern defines explicit handoffs=[...] on agents — a list of who CAN be handed off to. Our current design has origin_type='handoff' but no constraint on allowed targets. Whether we need an explicit "allowed handoff targets" on agent_definitions is a future question — probably lives in agent_subagents as a constraint.
