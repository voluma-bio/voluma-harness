meridian v1

Local-first coding-agent platform. Agents and humans co-edit files through CRDTs — edits merge by construction instead of conflicting. One TypeScript daemon, three surfaces, SQLite authority.

Differentiator: The write capability replaces native file-edit tools. Claude Code gets an MCP tool; subagents get a bash CLI; the webapp gets a live editor. All operate on the same Yjs documents. Nobody else ships merge-by-construction agent editing.
Python 0.x → TypeScript v1
Python proved multi-agent spawning, prompt packages, and multi-harness orchestration. It also hit the ceiling: 5 harness adapters, PTY hacks, JSONL scans, file-race repair. v1 replaces internals while keeping agent-facing contracts.
Shares flow packages
Not a fork. @meridian/agent-edit, @meridian/contracts, @meridian/prosemirror-schema, and @meridian/markup are npm library dependencies. meridian-flow stays the cloud product.
SQLite replaces files-as-state
WAL mode, Drizzle ORM, transactional writes, append-only journals. Crash-only spirit preserved: recovery IS startup.

CRDT write seam

Three bindings, one authority. The authority is WriteService, composed in the daemon from @meridian/agent-edit plus local adapters.

flowchart TD
  Agent --> Binding["Binding (MCP / CLI / in-process)"]
  Binding --> WS["Daemon WriteService"]
  WS --> Auth["1. authorize(capability, path)"]
  WS --> Core["2. AgentEditCore.write(command, context)"]
  Core --> YDoc["Y.Doc mutation"]
  Core --> Journal["SQLite journal append"]
  WS --> Proj["3. File projection refresh"]
  WS --> Outcome["WriteOutcome { status, text, writeId? }"]
  Outcome --> Return["Binding returns WriteOutcome.text"]

Binding 1: MCP tool (Claude Code primary)

Claude Code stays native for subscription auth and TUI. Deny Edit, Write, NotebookEdit in .claude/settings.json. Supply write as an MCP tool via .mcp.json (stdio transport).

Schema imports WriteCommandSchema directly from @meridian/agent-edit. Tool result is WriteOutcome.text — already LLM-readable, zero translation.

Binding 2: bash write CLI (subagents)

# JSON form (canonical)
write '{"command":"replace","file":"src/main.ts","find":"old","content":"new"}'

# Friendly flags (compile to same WriteCommand)
write replace --file src/main.ts --find "old" --content "new"
write undo --file src/main.ts --last 1

Reads context from env (MERIDIAN_SPAWN_ID, MERIDIAN_CAPABILITY_TOKEN, etc.), POSTs to daemon, prints WriteOutcome.text, exits 0/1/2.

Binding 3: in-process (API agents)

Embedded/API agents call WriteService directly inside the daemon. Same WriteCommandSchema, same outcome type.

File → CRDT projection

Cold open
Disk file parsed through code/prose codec → ProseMirror blocks → Y.Doc checkpoint if no journal exists.
Active session
Yjs authoritative. Agent writes + human edits mutate Y.Doc first, then refresh file projection.
External change
Watcher classifies as external edit. If Y.Doc active, import as concurrent update. If inactive, disk stays authority.
Large/binary files
Not tracked by default. write rejects binary or over-limit files unless explicit codec exists.

Harness architecture

v1 avoids rebuilding five native harness adapters. Two harness classes:

ClassExamplesWrite binding
Claude Code native claude -p --output-format stream-json MCP write + bash fallback
Unified API harness Anthropic API, OpenAI, Gemini, AI SDK In-process WriteService

Cursor, Pi, OpenCode, Codex native adapters are not v1 surface. They can stay in 0.x legacy or be represented through the unified API harness.

Spawn lifecycle (preserved)

queued → running → finalizing → succeeded
                 ↘               ↗
                  → failed / cancelled / timed_out

Terminal state is monotonic. Completion report evidence wins over late cancel signals.

Just-bash model for subagent writes

Advisory mode
Deny Edit/Write in settings, put write first on PATH, detect out-of-band changes as external edits.
Strict CRDT mode
OS sandbox with read-only source roots. Daemon write broker is the only writer. Writable dirs granted for temp/build.
Capability elevation
Commands that intentionally mutate outside CRDT (dep install, generated targets) need explicit grants, shown in UI/events.

Capability & security

Capability model

SpawnCapability {
  own: PermissionSet,        // what this spawn may do
  delegation: PermissionSet, // what its children may do
  max_depth: number          // depth limit
}

Invariant:
  child.own        ≤ parent.delegation
  child.delegation ≤ parent.delegation
  child.max_depth  ≤ parent.max_depth - 1

Enforcement points

BoundaryEnforcement
CLI → daemonCapability token required. Daemon loads stored capability, ignores caller claims.
MCP/CLI writeValidate token, actor, file grant, command grant, sandbox mode before path resolution.
Spawn creationParent delegation checked before row reservation. Blocked = depth/capability error, no side effects.
Harness launchEnvironment built from grants. Secrets and unknown MERIDIAN_* stripped.
SandboxOS strategy enforces file/command/network subset feasible on host.
WebappLocal user trusted. Remote binding requires explicit auth + project grants.

Sandbox modes

ModeMeaning
read-onlySource roots read-only via OS sandbox. Writes only through daemon write broker.
workspace-writeApproved workspace paths writable. CRDT tracked files still prefer write.
danger-full-accessNo filesystem containment. Capability enforcement still applies to daemon APIs.
noneUncontained local execution. Explicit & visible. For harnesses that can't be sandboxed.

Secrets

Secrets are grants, not ambient env inheritance. Default child env strips *_TOKEN, *_KEY, *_SECRET unless explicitly granted. Materialization to env vars is explicit per-grant.

Thread model

Threads are conversation/runtime objects. Spawns are process/lifecycle objects. A spawn normally owns one thread.

Cache invariant

System prompts are immutable after thread creation. This preserves provider prompt-cache behavior. Updated context flows through tool results and explicit read/search calls. When context drifts too far, create a new thread (hot-swap) with a fresh prompt — don't mutate the existing one.

Thread lifecycle

Created → Running → Stale (context drift) → HotSwapped → Closed
                  ↘                                         ↗
                   → Closed (completed or cancelled)

Thread creation steps

  1. Resolve config and mars launch bundle
  2. Resolve work/task/context roots
  3. Assemble system prompt (agent body + skills + invariants + capability instructions + context summary)
  4. Hash and store system_prompt_snapshot + context_snapshot_json
  5. Freeze prompt for thread lifetime
  6. Launch harness with thread IDs and capability context

Parent-child relationships (preserved)

MERIDIAN_CHAT_ID inherited across spawn tree. MERIDIAN_SPAWN_ID identifies current. MERIDIAN_PARENT_SPAWN_ID links parent. MERIDIAN_DEPTH zero-based, fail-closed. Tree views use stored parent_id, not env inspection.

Mars integration

mars-agents stays Rust and stays the prompt package manager. v1 does not reimplement package resolution, lock ownership, target lowering, model aliases, or launch-bundle construction.

Contract

  • meridian mars <args> passthrough with --root injection and MERIDIAN_MANAGED=1
  • Bundled/sibling mars binary resolution before PATH
  • mars build launch-bundle --json schema v3 as launch policy authority
  • Agent catalog from .mars/agents/, not harness-native target dirs
  • Mars exit codes 0–3 propagated

What simplifies

v1 only needs Claude Code native target materialization + canonical .mars/. Five harness targets in mars itself stay until mars changes — reducing configured targets is a template choice, not a meridian reimplementation.

Cloud sync (optional)

Local SQLite/Yjs remains authoritative when offline. Cloud sync is a peer, not a controller.

Sync scope

SyncableLocal-only
Yjs document updates/checkpoints, document metadata, thread/turn events, optional event summaries Secrets, raw env snapshots, absolute paths, sandbox/process details, private prompt paths

Protocol: meridian-flow's Yjs WebSocket (/ws/yjs prefix). Room names map to document IDs. The local daemon acts as a Yjs peer, not a remote database client.

Prior art synthesis

Patterns studied from 6 reference architectures, distilled to what v1 should steal.

Pi
Two-level provider seam (wire protocol vs vendor). Pure event-driven agent loop. I/O mode adapters over shared session.
OpenCode
Per-request instance routing via header/query. SSE with register-before-stream. Effect-TS + Drizzle on SQLite WAL.
Claude Agent SDK
NDJSON stream protocol. Permission interception hooks (canUseTool). Session store dual-write. Agent as declarative data.
CopilotKit
Runtime bridge between web UI and agent. Streaming via server-sent events. Frontend-registered actions as tools.
LangGraph
Graph-based orchestration with checkpoints. State-machine for control flow. Persistence adapters for multiple backends.
Omnigent
Meta-harness via executor.harness. Runner/server split. Stateful policy (cost caps, access controls). Child sessions as first-class objects.

Common patterns adopted

  • Authority daemon + thin clients (Pi, OpenCode, Omnigent) → our daemon + CLI/MCP/web model
  • Harness abstraction layer (Pi, Omnigent, AI SDK) → our two-class harness
  • SQLite WAL as local state (OpenCode) → our data model
  • Structured event stream (OpenCode SSE, Claude SDK NDJSON) → our event sink
  • Agent as declarative data (Claude SDK, Pi) → our mars agent catalog
  • Capability/policy enforcement at authority (Omnigent) → our capability model

Unique to v1

None of the studied systems have CRDT collaborative editing between human and agent. This is the product gap v1 fills.

Invariant registry

Implementation checklist. carry = preserve agent-facing behavior. redesign = behavior stays, implementation changes. break = intentionally dropped.

carry Agent-facing contracts
  • Spawn statuses: queued, running, finalizing, succeeded, failed, cancelled, timed_out
  • Spawn transition graph and terminal monotonicity
  • Completion report evidence wins over late cancel
  • p<N> spawn IDs and c<N> chat IDs — CLI-visible, monotonic
  • Parent/depth: parent_id, MERIDIAN_PARENT_SPAWN_ID, zero-based depth, fail-closed
  • MERIDIAN_CHAT_ID inherited across spawn tree
  • Work attachment precedence: explicit > ambient session > --from inheritance
  • Config merge: CLI flags > env > YAML/profile > config defaults
  • Core env exports: spawn, parent, project, depth, chat, task dir, context dirs
  • CLI command paths: spawn, session, work, context, mars, doctor
  • Spawn refs: @latest, @last-failed, @last-completed, p123
  • Fork policy: --fork identity-locked, --fork-fresh allows changes
  • Control protocol semantics: inject/interrupt/permission meanings preserved
  • Mars passthrough, launch-bundle v3, filesystem contract
redesign Internals changing
  • Runtime persistence: JSON files → SQLite tables (crash recovery preserved)
  • Harness layer: five adapters → Claude Code native + unified API
  • Terminal event extraction: explicit API events replace per-harness classifiers
  • Session transcripts: structured events/turns replace raw history.jsonl
  • Control socket: per-spawn sockets → daemon RPC
  • Observability: structlog → event sink + OTel
  • File authority: SQLite/Yjs/daemon replace files-as-authority for runtime
  • Agent-edit adapters: Postgres/Hocuspocus → local SQLite/in-process
  • Capability model: D8 audit → enforced runtime
break Intentionally dropped
  • Files-as-runtime-state (spawns/<id>/state.json)
  • history.jsonl as transcript authority
  • Native Cursor/Pi/OpenCode/Codex adapters
  • Global spawns.jsonl and legacy telemetry JSONL
  • Prompt body sidecar files → system prompt snapshots in SQLite
  • Per-harness target sprawl in default templates

Build order

Each step proves a vertical slice. The first step is the differentiator proof.

  1. Prove the write seam
    write CLI/MCP → daemon → in-memory AgentEditCore → stdout/tool-result. Validates CRDT write is viable end-to-end.
  2. SQLite journal + Y.Doc coordinator
    UpdateJournal/ReversalStore on SQLite, document lifecycle, in-process Y.Doc cache.
  3. File tracking + source codec
    Line-oriented codec with exact round-trip. Disk → Y.Doc → disk projection. Tree-sitter-aware blocks are a later optimization.
  4. Core spawn/session/thread/work schema
    SQLite tables, CLI daemon client, the full spawn lifecycle.
  5. Claude Code native launch
    Deny rules, MCP write tool, controlled env, process scope, stream parsing.
  6. Webapp
    Live editor + dashboard over the same daemon. TipTap/Yjs integration.
  7. Unified API harness
    In-process tools, embeddable agents, normalized events.
  8. Cloud Yjs peer sync
    Optional. Connect local daemon as Yjs peer to meridian-flow cloud.

Open questions

Decisions needing proof or a committed implementation choice before each relevant build step.

  • Q1 Package pin: Target current h/v3 agent-edit with SyncStateStore, or the draft-simplify successor with interactionContext and branch-peer docs? Recommendation: pin the successor before building adapters.
  • Q2 Web stack: TanStack Start, Vite+React, or other? Must: static assets from daemon, fast local TTI, Yjs/TipTap integration, Tailscale-hostable.
  • Q3 Unified API harness: Vercel AI SDK vs direct provider SDKs. Criteria: tool-call streaming, cancellation, usage accounting, provider escape hatches.
  • Q4 Sandbox depth: How far v1 goes on read-only roots, writable build dirs, mediated git/package-manager per OS. The seam is defined; feasibility needs probes.
  • Q5 Code codec: Start line-oriented or invest in tree-sitter blocks? Rec: line-oriented with round-trip tests, block IDs designed for later tree-sitter swap.
  • Q6 Daemon transport: HTTP default. If benchmarks miss <50ms, switch to UDS/named pipes without changing capability contracts.
  • Q7 Migration path: Big-bang v1 alongside 0.x, or importer for 0.x sessions/spawns? No real-user compat needed, but dogfooding may need import tooling.
  • Q8 Cloud metadata: Which thread/event/session metadata should meridian-flow store alongside Yjs updates? What auth model for Tailscale/cloud peers?
  • Q9 Work item authority: Is __status.json compatibility/export only, or co-authority for work status?
  • Q10 MCP hosting: Is apps/mcp-write always a daemon RPC client, or can it be in-process via future daemon-managed Claude transport?
