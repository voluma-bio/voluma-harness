# Emerging multi-agent contracts — source evidence

Evidence collected 2026-07-14. This is a source report, not a product recommendation.

## Agent-to-agent and agent-to-client interoperability

- **A2A (Google/Linux Foundation project).** The normative specification defines stateful `Task`s with `taskId`, `contextId`, lifecycle states, `Message` and `Artifact` objects made of typed `Part`s, and optional extensions. A request may return a task immediately and continue asynchronously; clients can poll, consume ordered SSE events, or configure webhook push notifications. Streams begin with a `Task`/`Message`, then status/artifact updates, and close at terminal state. Agents publish an `AgentCard` with capabilities (streaming, push notifications, extended card) and extension URIs; unsupported capabilities must produce protocol errors. This reserves stable IDs, lifecycle, ordered event, capability-negotiation, artifact, and extension fields, while making delivery/reconnect, auth, duplicate events, and cross-trust-boundary authorization explicit risks. Sources: [spec](https://github.com/a2aproject/A2A/blob/main/docs/specification.md), [protobuf schema](https://github.com/a2aproject/A2A/blob/main/specification/a2a.proto), repository [README](https://github.com/a2aproject/A2A).
- **ACP (Agent Client Protocol).** The official repository says ACP standardizes communication between interactive code editors and coding agents. Wire compatibility is negotiated during `initialize` using `protocolVersion`; optional behavior is selected through capabilities, not crate/schema package version. The repo ships JSON schemas, Rust/Python/TypeScript SDK integrations and examples, and states stable protocol version `1`. This is concrete evidence of an editor↔agent contract rather than agent↔agent semantics; reserve negotiated version/capabilities and structured session/update messages, and do not assume ACP supplies shared multi-agent coordination. Sources: [README](https://github.com/agentclientprotocol/agent-client-protocol), [schema directory](https://github.com/agentclientprotocol/agent-client-protocol/tree/main/schema), [GitHub Copilot ACP server documentation](https://docs.github.com/en/enterprise-cloud@latest/copilot/reference/copilot-cli-reference/acp-server).

## Multiplayer/shared sessions

- A2A explicitly permits additional messages on non-terminal tasks for multi-turn interaction, and its `contextId` groups related tasks. The contract is task/context based, not a shared-memory or conflict-resolution protocol. Evidence: [A2A specification, task/context and multi-turn sections](https://github.com/a2aproject/A2A/blob/main/docs/specification.md).
- LangGraph separates a thread (accumulated state for a sequence of runs) from a cross-thread `Store`; stores are namespaced tuples and key/value records, with production backends listed as Postgres, MongoDB, and Redis. This demonstrates a useful split between run/session state and durable knowledge, while exposing consistency, tenancy, deletion, embedding/index drift, and concurrent-writer risks. Source: [persistence and memory docs](https://docs.langchain.com/oss/python/langgraph/persistence), [memory docs](https://docs.langchain.com/oss/python/langgraph/add-memory).

## Deterministic trace/replay and evaluation

- LangGraph checkpoints graph state at every super-step, records state snapshots with `thread_id`, `checkpoint_id`, parent, metadata, writes, step, and tasks, and exposes state history/time travel. Interrupts persist state and resume without rerunning successful nodes. However, the documented compatibility behavior says resumed threads immediately use the latest graph code, unlike engines that pin a run to its starting code; this is a replay/reproducibility risk. Sources: [persistence](https://docs.langchain.com/oss/python/langgraph/persistence), [interrupts](https://langchain-ai.github.io/langgraph/concepts/breakpoints/), [compatibility](https://docs.langchain.com/oss/python/langgraph/backward-compatibility).
- LangSmith's run-data contract includes run/session identifiers, execution metadata, token usage and `total_cost`; its evaluation product supports datasets, human annotation, heuristic checks, LLM-as-judge, and pairwise comparison. Reserve immutable run/span IDs, parent-child relationships, input/output, model/config, usage/cost, timestamps, evaluator results, and dataset/example/version references. Sources: [run data format](https://docs.langchain.com/langsmith/run-data-format), [cost tracking](https://docs.langchain.com/langsmith/cost-tracking), [evaluation](https://www.langchain.com/langsmith/evaluation).
- OpenAI Evals is an active open-source framework and benchmark registry with custom eval definitions and JSONL sample data under `evals/registry/{evals,data}`. Its README notes private evals are supported; repository instructions also note registry data may use Git LFS. This is evidence for versioned datasets, evaluator implementations, sample cases, and local/CI execution—not evidence of deterministic agent replay. Sources: [repository](https://github.com/openai/evals), [build-eval guide](https://github.com/openai/evals/blob/main/docs/build-eval.md), [custom eval guide](https://github.com/openai/evals/blob/main/docs/custom-eval.md).

## Budget and cost governance

- LangSmith automatically records provider token usage/cost and accepts custom cost data for other components; run data exposes `total_cost`. This reserves per-span usage and cost fields and an aggregation boundary at trace/session/project. It does not establish authorization budgets, hard stops, or billing truth; provider pricing changes, missing usage, retries, parallel branches, and custom tools remain risks. Sources: [cost tracking](https://docs.langchain.com/langsmith/cost-tracking), [run data format](https://docs.langchain.com/langsmith/run-data-format).
- A2A capability negotiation and structured error model provide protocol seams for refusing unsupported operations, but the specification evidence reviewed does not define a spend/quota budget model. Source: [A2A specification](https://github.com/a2aproject/A2A/blob/main/docs/specification.md).

## Durable memory/knowledge layers

- LangGraph's documented model has short-term, thread-level checkpoint memory and long-term cross-thread Store memory. Store records have namespace, key, value, and optional indexed fields; production persistence is delegated to backend integrations. Reserve scope/namespace, record identity, provenance/created-at/version, access policy, and explicit retrieval/index metadata. Risks include full-state checkpoint growth (the docs call out storage growth and describe beta `DeltaChannel`), stale or unsafe memories, and semantic-index rebuild/versioning. Source: [memory and persistence](https://docs.langchain.com/oss/python/langgraph/persistence).

## Maturity/adoption signals (descriptive only)

- ACP's official GitHub page reports published schemas and SDKs plus a stable wire protocol version 1; GitHub metadata shown on the repository reports 3.2k stars, 261 forks, and 45 releases (latest listed May 22, 2026). Treat repository metrics as signals, not conformance or production reliability. Source: [ACP repository](https://github.com/agentclientprotocol/agent-client-protocol).
- A2A maintains a normative specification and protobuf schema in its public repository; LangGraph/LangSmith publish detailed operational contracts; OpenAI Evals publishes a registry and contribution structure. None of these sources alone proves interoperability across independent implementations or guarantees deterministic LLM outputs.
