Harness architecture

v1 avoids rebuilding five native harness adapters. Two harness classes cover all providers.

Interfaces

Both classes present the same lifecycle handle to the daemon. Provider-specific launch and event parsing stay behind this boundary.

TypeScript
interface AgentHarness {
  id: "claude-code" | "api";
  launch(input: HarnessLaunchInput): Promise<HarnessRunHandle>;
}

interface HarnessRunHandle {
  spawnId: string;
  interrupt(message?: string): Promise<void>;
  inject(input: UserInjection): Promise<void>;
  stop(reason: string): Promise<void>;
  events(): AsyncIterable<HarnessEvent>;
}

interface HarnessEvent {
  seq?: number;
  spawnId: string;
  threadId: string;
  turnId?: string;
  eventType: string;
  harnessId: "claude-code" | "api";
  payload: unknown;
  rawText?: string;
  observedAt: string;
}

Spawn lifecycle

stateDiagram-v2 [*] --> queued queued --> running queued --> cancelled queued --> timed_out running --> finalizing running --> failed running --> cancelled running --> timed_out finalizing --> succeeded finalizing --> failed finalizing --> cancelled finalizing --> timed_out

Terminal state is monotonic. Completion report evidence wins over late cancel signals.

Claude Code native launch steps

  1. Resolve launch bundle from mars
  2. Assemble and freeze the thread system prompt
  3. Create spawn/thread rows before process launch
  4. Create narrowed capability and token
  5. Inject environment, controlled PATH, MCP config, and deny rules
  6. Spawn Claude Code under a platform process scope
  7. Persist harness stream events before observing and fan-out
  8. Observe terminal result, then finalize

Just-bash model

Advisory mode

Deny Edit/Write in settings, put write first on PATH, and detect out-of-band changes as external edits.

Strict CRDT mode

OS sandbox with read-only source roots. The daemon write broker is the only writer; writable directories are granted for temp and build output.

Capability elevation

Commands that intentionally mutate outside CRDT, such as dependency installs or generated targets, need explicit grants surfaced in UI and events.

Workspace resolution

The simplified v1 model replaces the Python 0.x six-level cascade with three deterministic levels.

  1. Explicit --workspace on the spawn command
  2. Work item's default workspace
  3. Project root, as the fallback

At creation: The daemon resolves the workspace, stores it in the spawn row, and sets the process cwd. No environment propagation is needed.
