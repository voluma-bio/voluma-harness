CRDT Write Seam

Three bindings, one authority. The authority is WriteService, composed in the daemon from @voluma/agent-edit plus local adapters. All agent writes land in the same Yjs document, so human and agent edits merge by construction.

One write path

Bindings change how an agent reaches the daemon, not the behavior of the write. Authorization, CRDT mutation, journaling, projection, and LLM-readable feedback stay behind WriteService.

Write flow

Every agent-facing route resolves to the same daemon service. The journal records the mutation before the projected file refreshes, and callers receive a concise outcome instead of transport-specific data.

Three bindings

Each binding keeps its host environment native while preserving the same command schema and result contract.

01

MCP tool, Claude Code primary

  • Claude Code stays native for subscription authentication and its TUI.
  • Deny Edit, Write, and NotebookEdit in .claude/settings.json.
  • Supply write as an MCP tool through .mcp.json over stdio.
  • The tool schema imports WriteCommandSchema from @voluma/agent-edit.
  • The tool result is WriteOutcome.text, already LLM-readable and requiring no translation.
02

bash write CLI, subagents

The CLI accepts the canonical JSON command or a friendly flag form, reads spawn context from the environment, POSTs to the daemon, prints WriteOutcome.text, and exits 0, 1, or 2.

# JSON form (canonical)
write '{"command":"replace","file":"src/main.ts","find":"old","content":"new"}'

# Friendly flags
write replace --file src/main.ts --find "old" --content "new"
write undo --file src/main.ts --last 1

MERIDIAN_SPAWN_ID and MERIDIAN_CAPABILITY_TOKEN supply the invocation context.

03

In-process, API agents

Embedded and API agents call WriteService directly inside the daemon. They use the same schema and receive the same outcome as MCP and CLI callers.

The service surface stays small: six commands, one file target, and optional locality controls for precise edits.

File → CRDT projection

A file only becomes collaborative through a codec and an active Yjs document. The projection keeps on-disk tools in the loop without letting disk writes bypass the authority.

Cold open
A disk file is parsed through a code or prose codec into ProseMirror blocks, then checkpointed into the Y.Doc when no journal exists.
Active session
Yjs is authoritative. Agent writes and human edits mutate the Y.Doc first; the file projection refreshes afterward.
External change
The watcher classifies an external edit. With an active Y.Doc, it imports a concurrent update; when inactive, disk remains the authority.
Large or binary files
These files are not tracked by default. write rejects binary or over-limit files unless an explicit codec exists.
